# Quản lý Vận hành SaaS (SaaS運用管理) - Đặc tả Nghiệp vụ

## 1. Tổng quan

### 1.1 Mục đích module

Module Quản lý Vận hành SaaS chịu trách nhiệm quản lý toàn bộ hoạt động vận hành (operations) của nền tảng Carima Staffing sau khi tenant đã được khởi tạo và đưa vào sử dụng. Module này tập trung vào các khía cạnh:

- **Giám sát hệ thống** (System Monitoring): theo dõi tình trạng sức khỏe, hiệu năng và tài nguyên của toàn bộ platform và từng tenant
- **Vận hành vòng đời tenant** (Tenant Lifecycle Operations): tạm dừng, phục hồi, sao lưu, di chuyển, ngừng hoạt động tenant
- **Bảo trì platform** (Platform Maintenance): bảo trì định kỳ, triển khai phiên bản mới, quản lý cấu hình và feature flag
- **Phân tích sử dụng** (Usage Analytics): theo dõi số nhân sự hoạt động cho billing, thống kê sử dụng hệ thống
- **Quản lý sự cố** (Incident Management): ghi log lỗi, cảnh báo, xử lý sự cố, giám sát SLA
- **Vận hành OEM/White-label**: quản lý cấu hình thương hiệu, báo cáo theo Provider
- **Quản lý batch job**: giám sát, xử lý lỗi, quản lý các tác vụ xử lý hàng loạt

> **Phạm vi:** Module này tập trung vào VẬN HÀNH LIÊN TỤC sau khi tenant đã được thiết lập. Việc khởi tạo tenant và onboarding được quản lý tại Module 02 — Quản lý Tenant.
>

## 3. Luồng nghiệp vụ chính

### 3.1 Luồng giám sát và xử lý sự cố

```
[Hệ thống]                          [Platform Admin]            [Platform Operator]
     |                                   |                           |
     |-- (1) Health check tự động ------>|                           |
     |   (mỗi 1 phút)                   |                           |
     |                                   |                           |
     |-- (2) Thu thập metric ----------->|                           |
     |   (CPU, RAM, DB, API latency)     |                           |
     |                                   |                           |
     |-- (3) Phát hiện bất thường ------>|                           |
     |   (vượt ngưỡng cảnh báo)         |                           |
     |                                   |                           |
     |                                   |-- (4) Xem dashboard ----->|
     |                                   |   sức khỏe hệ thống      |
     |                                   |                           |
     |-- (5) Tự động phục hồi --------->|                           |
     |   (restart service, failover)     |                           |
     |                                   |                           |
     |   [Nếu tự động phục hồi thất bại]|                           |
     |                                   |                           |
     |-- (6) Leo thang cảnh báo -------->|                           |
     |                                   |-- (7) Tạo incident ------>|
     |                                   |                           |
     |                                   |                    (8) Xử lý kỹ thuật
     |                                   |                    (manual intervention)
     |                                   |                           |
     |                                   |<--- (9) Cập nhật ----------|
     |                                   |   trạng thái incident      |
     |                                   |                           |
     |-- (10) Ghi Audit Log ------------>|                           |
```

**Mô tả chi tiết:**

1. Hệ thống thực hiện health check tự động mỗi 1 phút cho tất cả các service (ECS tasks, RDS, Elasticache, SQS)
2. Thu thập metric hiệu năng và tài nguyên từ CloudWatch, gửi về dashboard
3. Khi metric vượt ngưỡng (error rate > 1%, latency p95 > 3s, CPU > 80%), hệ thống tạo cảnh báo
4. Platform Admin xem tổng quan trạng thái trên dashboard sức khỏe
5. Hệ thống thử tự động phục hồi: restart ECS task bị lỗi, failover RDS nếu cần
6. Nếu tự động phục hồi thất bại hoặc sự cố nghiêm trọng, hệ thống leo thang cảnh báo (email + Slack)
7. Platform Admin tạo incident ticket để theo dõi xử lý
8. Platform Operator can thiệp xử lý kỹ thuật
9. Cập nhật trạng thái incident: Investigating → Identified → Resolved
10. Toàn bộ quy trình được ghi Audit Log

### 4.1 Dashboard sức khỏe hệ thống (F-1401)

- **Actor:** Platform Admin
- **Màn hình:** Platform Admin Portal > Operations > System Health
- **Phase:** BF-M09
- **Mô tả:** Dashboard tổng hợp hiển thị trạng thái sức khỏe của toàn bộ platform Carima Staffing. Cung cấp cái nhìn tổng thể về tình trạng các thành phần hạ tầng, giúp Platform Admin nhanh chóng phát hiện và xử lý sự cố.
- **Điều kiện tiên quyết:**
    - Platform Admin đã đăng nhập với quyền quản trị platform
- **Thành phần dashboard:**
    
    **Panel 1 — Trạng thái tổng thể:**
    
    | Thành phần | Chỉ số giám sát | Nguồn dữ liệu |
    | --- | --- | --- |
    | ECS Fargate (API) | Số task đang chạy, CPU/Memory trung bình, số task bị lỗi | CloudWatch ECS Metrics |
    | RDS PostgreSQL | Trạng thái instance, connections, replication lag | CloudWatch RDS Metrics |
    | Elasticache (Redis) | Hit rate, memory usage, connections | CloudWatch ElastiCache Metrics |
    | SQS Queues | Queue depth, message age, dead-letter queue count | CloudWatch SQS Metrics |
    | S3 Storage | Tổng dung lượng, số object, request rate | CloudWatch S3 Metrics |
    | CloudFront | Request count, error rate, cache hit ratio | CloudWatch CF Metrics |
    | SES (Email) | Bounce rate, complaint rate, send quota remaining | CloudWatch SES Metrics |
    
    **Panel 2 — Chỉ số sức khỏe tổng hợp:**
    
    | Chỉ số | Mức xanh (OK) | Mức vàng (Warning) | Mức đỏ (Critical) |
    | --- | --- | --- | --- |
    | API Error Rate | < 0.5% | 0.5% - 1% | > 1% |
    | API Latency (p95) | < 2s | 2s - 3s | > 3s |
    | DB Connection Usage | < 60% | 60% - 80% | > 80% |
    | CPU Usage (ECS) | < 60% | 60% - 80% | > 80% |
    | Memory Usage (ECS) | < 70% | 70% - 85% | > 85% |
    | Disk Usage (RDS) | < 60% | 60% - 80% | > 80% |
    | SQS Dead Letter Queue | 0 | 1 - 10 | > 10 |
    | SES Bounce Rate | < 2% | 2% - 5% | > 5% |
    
    **Panel 3 — Lịch sử sự cố gần đây:**
    
    - 10 sự cố gần nhất với trạng thái (Open / Investigating / Resolved)
    - Thời gian phát hiện, thời gian xử lý
    
    **Panel 4 — Uptime SLA:**
    
    - Uptime tháng hiện tại (%)
    - Uptime 12 tháng gần nhất (biểu đồ cột)
    - Số giờ downtime còn lại trong SLA (99.9%)
- **Tự động làm mới:** Dashboard tự động cập nhật mỗi 30 giây
- **Quy tắc nghiệp vụ:**
    - BR-140: Dashboard sức khỏe chỉ hiển thị cho Platform Admin, không hiển thị cho tenant admin
    - BR-141: Khi bất kỳ chỉ số nào ở mức ĐỎ (Critical), hệ thống tự động gửi cảnh báo qua email và Slack cho Platform Admin
- **Tham chiếu:** Module 13 (NFR-20 — Giám sát và cảnh báo)

### 4.2 Giám sát sức khỏe tenant (F-1402)

- **Actor:** Platform Admin
- **Màn hình:** Platform Admin Portal > Operations > Tenant Health
- **Phase:** BF-M09
- **Mô tả:** Giám sát trạng thái sức khỏe của từng tenant riêng biệt: trạng thái database, dung lượng lưu trữ, số lượng user hoạt động, và các chỉ số quan trọng khác.
- **Điều kiện tiên quyết:**
    - Platform Admin đã đăng nhập
- **Luồng xử lý chính:**
    1. Platform Admin truy cập màn hình “Tenant Health”
    2. Hệ thống hiển thị danh sách tất cả tenant với trạng thái sức khỏe:
        
        
        | Trường | Mô tả |
        | --- | --- |
        | Tên tenant | Tên công ty |
        | Loại | MOTO / SAKI |
        | Trạng thái | ACTIVE / SUSPENDED / DECOMMISSIONING |
        | DB Status | OK / Warning / Error |
        | DB Size | Dung lượng database hiện tại (MB/GB) |
        | S3 Storage | Dung lượng file lưu trữ (MB/GB) |
        | Active Users | Số user đăng nhập trong 30 ngày gần nhất |
        | Last Activity | Thời điểm hoạt động cuối cùng |
        | Sức khỏe tổng hợp | Xanh / Vàng / Đỏ |
    3. Click vào tenant để xem chi tiết:
        - Biểu đồ sử dụng DB theo thời gian
        - Danh sách user và trạng thái hoạt động
        - Số lượng record theo từng bảng chính (hợp đồng, chấm công, hóa đơn)
        - Connection pool usage của tenant
        - Slow query gần đây của tenant
    4. Có thể lọc theo: trạng thái sức khỏe, loại tenant, Provider
- **Quy tắc nghiệp vụ:**

### 4.4 Giám sát tài nguyên theo tenant (F-1404)

### 4.5 Tạm dừng / Tái kích hoạt tenant — Chi tiết vận hành (F-1405)

- **Actor:** Platform Admin
- **Màn hình:** Platform Admin Portal > Tenant Ops > Suspension
- **Phase:** BF-M09
- **Mô tả:** Bổ sung chi tiết vận hành cho chức năng tạm dừng/tái kích hoạt tenant đã định nghĩa tại Module 02 (F-0208). Module này tập trung vào khía cạnh kỹ thuật và quy trình vận hành.

> **Lưu ý:** Luồng nghiệp vụ cơ bản (lý do tạm dừng, xác nhận, chuyển trạng thái) đã được định nghĩa tại Module 02 — F-0208. Phần này bổ sung các quy trình vận hành kỹ thuật.
> 
- **Quy trình kỹ thuật khi tạm dừng:**
    1. Hệ thống vô hiệu hóa tất cả session đang hoạt động của tenant (force logout)
    2. Hệ thống cập nhật connection pool: đóng tất cả connection của tenant
    3. Hệ thống vô hiệu hóa tất cả EventBridge rule của tenant (dừng batch job)
    4. Hệ thống đặt database schema của tenant sang read-only mode
    5. Hệ thống lưu snapshot database trước khi tạm dừng
    6. Hệ thống gửi email thông báo cho tất cả admin của tenant
    7. Ghi Audit Log chi tiết bao gồm: số user bị force logout, số batch job bị dừng
- **Quy trình kỹ thuật khi tái kích hoạt:**
    1. Hệ thống xác nhận database schema vẫn nguyên vẹn (integrity check)
    2. Hệ thống chuyển database schema từ read-only về read-write mode
    3. Hệ thống kích hoạt lại EventBridge rule của tenant
    4. Hệ thống mở lại connection pool cho tenant
    5. Hệ thống chạy batch catch-up cho các job đã bị dừng (nếu cần)
    6. Hệ thống gửi email thông báo tái kích hoạt cho tất cả admin
    7. Ghi Audit Log
- **Quy tắc nghiệp vụ:**
    - BR-14A: Trước khi tạm dừng, hệ thống PHẢI tạo snapshot database tự động (để đảm bảo có thể phục hồi nếu cần)
    - BR-14B: Khi tái kích hoạt, hệ thống phải chạy integrity check trước khi mở truy cập
    - BR-14C: Batch job bị dừng do tạm dừng phải được đánh dấu và xử lý catch-up khi tái kích hoạt
- **Tham chiếu:** Module 02 (F-0208, BR-020L đến BR-020O)

### 4.6 Sao lưu và phục hồi dữ liệu tenant (F-1406)

- **Actor:** Platform Admin, Platform Operator
- **Màn hình:** Platform Admin Portal > Tenant Ops > Backup/Restore
- **Phase:** BF-M09
- **Mô tả:** Quản lý quy trình sao lưu (backup) và phục hồi (restore) dữ liệu cho từng tenant độc lập. Hỗ trợ cả sao lưu tự động theo lịch và sao lưu thủ công khi cần.
- **Điều kiện tiên quyết:**
    - Platform Admin hoặc Platform Operator đã đăng nhập
    - Tenant cần sao lưu/phục hồi đang ở trạng thái ACTIVE hoặc SUSPENDED
- **Luồng xử lý chính — Sao lưu thủ công:**
    1. Platform Admin chọn tenant cần sao lưu
    2. Chọn loại sao lưu:
        - Full backup (database + S3 files)
        - Database only
        - Files only (S3)
    3. Nhập ghi chú cho bản sao lưu (tùy chọn)
    4. Xác nhận thực hiện
    5. Hệ thống bắt đầu quy trình sao lưu (background job)
    6. Hệ thống hiển thị tiến trình và thông báo khi hoàn tất
    7. Ghi Audit Log: “Manual backup created for tenant {tenant_name}”
- **Luồng xử lý chính — Phục hồi:**
    1. Platform Admin chọn tenant cần phục hồi
    2. Xem danh sách các bản sao lưu khả dụng:
        
        
        | Trường | Mô tả |
        | --- | --- |
        | Backup ID | Mã định danh bản sao lưu |
        | Thời điểm | Ngày giờ tạo sao lưu (JST) |
        | Loại | Full / Database / Files |
        | Kích thước | Dung lượng (GB) |
        | Trạng thái | Completed / In Progress / Failed |
        | Ghi chú | Mô tả do người tạo nhập |
    3. Chọn bản sao lưu cần phục hồi
    4. Chọn chế độ phục hồi:
        - **Phục hồi ghi đè (Overwrite):** Thay thế toàn bộ dữ liệu hiện tại
        - **Phục hồi song song (Clone):** Tạo schema mới để kiểm tra trước khi áp dụng
    5. Xác nhận 2 lần (dialog xác nhận + nhập mã xác nhận)
    6. Hệ thống tạm dừng tenant (nếu đang ACTIVE) trước khi phục hồi
    7. Hệ thống thực hiện phục hồi
    8. Hệ thống chạy integrity check sau phục hồi
    9. Hệ thống tái kích hoạt tenant (nếu trước đó đang ACTIVE)
    10. Ghi Audit Log: “Database restored for tenant {tenant_name} from backup {backup_id}”
- **Sao lưu tự động:**
    
    
    | Loại | Tần suất | Thời gian | Retention |
    | --- | --- | --- | --- |
    | Daily snapshot | Hàng ngày | 02:00 - 04:00 JST (off-peak) | 30 ngày (mặc định, cấu hình theo tenant) |
    | Point-in-time recovery | Liên tục | — | 7 ngày |
    | Weekly full backup | Hàng tuần (Chủ nhật) | 01:00 - 05:00 JST | 90 ngày |
    | Monthly archive | Hàng tháng (ngày 1) | 00:00 - 06:00 JST | 1 năm |
- **Luồng ngoại lệ:**
    - **E-1406-1:** Sao lưu thất bại (RDS error, S3 error) → Ghi log lỗi, retry tự động tối đa 3 lần, gửi cảnh báo cho Platform Admin
    - **E-1406-2:** Phục hồi thất bại giữa chừng → Rollback về trạng thái trước khi phục hồi, gửi cảnh báo
    - **E-1406-3:** Không đủ dung lượng S3 để lưu sao lưu → Cảnh báo Platform Admin, đề xuất xóa các bản sao lưu cũ
- **Quy tắc nghiệp vụ:**
    - BR-14D: Mỗi tenant có lịch sao lưu độc lập, có thể tùy chỉnh retention period (tham chiếu BR-024)
    - BR-14E: Phục hồi dữ liệu (restore) PHẢI yêu cầu xác nhận 2 lần vì đây là thao tác không thể hoàn tác
    - BR-14F: Trước khi phục hồi ghi đè, hệ thống PHẢI tự động tạo snapshot của dữ liệu hiện tại (để có thể rollback)
    - BR-14G: Sao lưu của mỗi tenant được lưu trữ tách biệt trên S3 (prefix riêng theo tenant_id)
    - BR-14H: RTO/RPO của mỗi tenant phải tuân theo cấu hình tại Module 02 (F-0202) và NFR-18
- **Tham chiếu:** Module 02 (BR-024), Module 13 (NFR-18, NFR-22)

### 4.7 Di chuyển database tenant — Scaling (F-1407)

- **Actor:** Platform Operator
- **Màn hình:** Platform Admin Portal > Tenant Ops > Migration
- **Phase:** BF-M09
- **Mô tả:** Khi tenant tăng trưởng vượt khả năng của RDS instance hiện tại, Platform Operator thực hiện di chuyển database sang instance lớn hơn hoặc instance riêng. Áp dụng cho Phase 2 và Phase 3 khi số lượng tenant tăng lên.
- **Điều kiện tiên quyết:**
    - Phải có kế hoạch di chuyển đã được phê duyệt
    - Phải thông báo trước cho tenant admin (tối thiểu 72 giờ)
    - Phải có bản sao lưu mới nhất trước khi bắt đầu
- **Kịch bản di chuyển:**
    
    
    | Kịch bản | Mô tả | Áp dụng |
    | --- | --- | --- |
    | Scale-up RDS | Tăng instance class (vd: db.r5.large → db.r5.xlarge) | Khi 1 instance phục vụ nhiều tenant cần nhiều tài nguyên hơn |
    | Tenant isolation | Di chuyển 1 tenant từ shared RDS sang dedicated RDS | Khi tenant lớn cần hiệu năng riêng biệt |
    | Rebalancing | Phân bổ lại các tenant giữa nhiều RDS instance | Khi tải không cân bằng giữa các instance |
- **Luồng xử lý chính (Tenant Isolation):**
    1. Platform Operator chọn tenant cần di chuyển
    2. Hệ thống hiển thị thông tin hiện tại: dung lượng DB, số connection, throughput
    3. Platform Operator chọn RDS instance đích (hoặc tạo mới)
    4. Hệ thống tạo kế hoạch di chuyển:
        1. Ước tính thời gian di chuyển
        2. Ước tính thời gian downtime (mục tiêu: < 5 phút)
        3. Tạo checklist kiểm tra
    5. Nhấn “Bắt đầu di chuyển”
    6. Hệ thống thực hiện:
        1. Tạo replica của schema trên instance đích
        2. Đồng bộ dữ liệu (initial sync)
        3. Bắt đầu replication liên tục (CDC — Change Data Capture)
        4. Khi replication lag < 1 giây: chuyển sang chế độ cutover
        5. Tạm dừng write trên source (maintenance mode cho tenant)
        6. Đợi replication bắt kịp (< 5 phút)
        7. Cập nhật connection config trỏ sang instance đích
        8. Mở lại write trên instance đích
        9. Xác nhận dữ liệu nguyên vẹn (checksum)
    7. Hệ thống chạy smoke test tự động
    8. Ghi Audit Log: “Tenant {tenant_name} migrated from {source} to {target}”
- **Luồng ngoại lệ:**
    - **E-1407-1:** Replication gap không giảm → Dừng quy trình, thông báo Platform Operator
    - **E-1407-2:** Checksum không khớp sau di chuyển → Rollback về instance cũ, cảnh báo
- **Quy tắc nghiệp vụ:**
    - BR-14I: Di chuyển database phải đảm bảo downtime tối đa < 5 phút cho tenant bị ảnh hưởng
    - BR-14J: Phải tạo full backup trước mỗi lần di chuyển (bắt buộc)
    - BR-14K: Sau khi di chuyển thành công, dữ liệu trên instance cũ được giữ lại 7 ngày trước khi xóa
- **Tham chiếu:** Module 13 (NFR-03a, NFR-03b — Horizontal scaling, Read replica)

### 4.8 Ngừng hoạt động tenant — Decommission (F-1408)

- **Actor:** Platform Admin
- **Màn hình:** Platform Admin Portal > Tenant Ops > Decommission
- **Phase:** BF-M09
- **Mô tả:** Quy trình ngừng vĩnh viễn hoạt động của một tenant, bao gồm lưu trữ dữ liệu theo yêu cầu pháp luật, dọn dẹp tài nguyên, và xóa schema khi hết thời hạn lưu trữ.
- **Điều kiện tiên quyết:**
    - Tenant đã ở trạng thái SUSPENDED (không thể ngừng trực tiếp từ ACTIVE)
    - Không có giao dịch đang xử lý (hợp đồng đang trong luồng phê duyệt)
    - Platform Admin đã xác nhận với tenant admin về việc ngừng hoạt động
- **Luồng xử lý chính:**
    1. Platform Admin truy cập “Ngừng hoạt động tenant”
    2. Hệ thống kiểm tra điều kiện tiên quyết và hiển thị checklist:
        
        
        | # | Hạng mục kiểm tra | Trạng thái |
        | --- | --- | --- |
        | 1 | Tenant đã ở trạng thái SUSPENDED | OK / NG |
        | 2 | Không có hợp đồng đang xử lý | OK / NG (số hợp đồng còn) |
        | 3 | Không có hóa đơn chưa thanh toán (phí sử dụng) | OK / NG (số hóa đơn còn) |
        | 4 | Dữ liệu đã được export cho tenant | OK / Chưa |
        | 5 | Xác nhận bằng văn bản từ tenant | OK / Chưa |
    3. Platform Admin xác nhận tất cả hạng mục
    4. Hệ thống yêu cầu nhập lý do ngừng hoạt động
    5. Xác nhận 2 lần (dialog + nhập mã xác nhận)
    6. Hệ thống thực hiện quy trình ngừng hoạt động:
        
        **Giai đoạn 1 — Lưu trữ dữ liệu (ngay lập tức):**
        
        1. Tạo full backup cuối cùng
        2. Archive toàn bộ dữ liệu sang S3 Glacier (định dạng dump + metadata)
        3. Archive toàn bộ file S3 của tenant sang Glacier
        
        **Giai đoạn 2 — Dọn dẹp kỹ thuật (sau khi archive hoàn tất):**
        
        1. Xóa tất cả EventBridge rule của tenant
        2. Xóa tất cả SQS queue riêng của tenant (nếu có)
        3. Xóa các cache key của tenant trên Redis
        4. Chuyển trạng thái tenant sang DECOMMISSIONED
        
        **Giai đoạn 3 — Xóa dữ liệu (sau thời hạn lưu trữ pháp lý):**
        
        1. Xóa database schema của tenant
        2. Xóa file S3 của tenant (trừ archive trên Glacier)
        3. Xóa bản ghi archive trên Glacier (sau 7 năm)
    7. Ghi Audit Log: “Tenant {tenant_name} decommissioned. Archive ID: {archive_id}”
- **Thời hạn lưu trữ sau khi ngừng:**
    
    
    | Loại dữ liệu | Thời hạn lưu trữ | Căn cứ |
    | --- | --- | --- |
    | Dữ liệu chấm công | 5 năm | 労働基準法 |
    | Dữ liệu hợp đồng | 5 - 7 năm | 派遣法 |
    | Tài liệu hóa đơn | 7 năm | 税法 + 電帳法 |
    | Audit log | 7 năm | Chính sách nội bộ |
    | Dữ liệu khác | 1 năm | Chính sách nội bộ |
- **Luồng ngoại lệ:**
    - **E-1408-1:** Archive thất bại → Dừng quy trình, không xóa dữ liệu, gửi cảnh báo
    - **E-1408-2:** Còn hóa đơn chưa thanh toán → Không cho phép ngừng hoạt động cho đến khi xử lý xong
- **Quy tắc nghiệp vụ:**
    - BR-14L: Tenant PHẢI ở trạng thái SUSPENDED trước khi có thể ngừng hoạt động (không được ngừng trực tiếp từ ACTIVE)
    - BR-14M: Dữ liệu PHẢI được archive hoàn tất trước khi bất kỳ thao tác xóa nào được thực hiện
    - BR-14N: Thời hạn lưu trữ dữ liệu theo pháp luật Nhật Bản là bắt buộc, không thể rút ngắn
    - BR-14O: Sau khi ngừng hoạt động, domain riêng (nếu là Provider) phải được giải phóng để có thể tái sử dụng
- **Tham chiếu:** Module 13 (NFR-21, NFR-22 — Data Retention)

### 4.9 Quản lý khung giờ bảo trì (F-1409)

- **Actor:** Platform Admin
- **Màn hình:** Platform Admin Portal > Maintenance > Schedule
- **Phase:** BF-M09
- **Mô tả:** Lên lịch và quản lý các đợt bảo trì hệ thống định kỳ. Đảm bảo việc bảo trì được thông báo trước cho người dùng và thực hiện trong khung giờ ít ảnh hưởng nhất.

### 4.10 Triển khai không gián đoạn — Zero-downtime Deployment (F-1410)

- **Actor:** Platform Operator
- **Màn hình:** Platform Admin Portal > Maintenance > Deployment
- **Phase:** BF-M09
- **Mô tả:** Quy trình triển khai phiên bản mới của ứng dụng mà không gây gián đoạn dịch vụ. Sử dụng cơ chế rolling update trên ECS Fargate và blue-green deployment cho các thay đổi lớn.
- **Điều kiện tiên quyết:**
    - Phiên bản mới đã được kiểm thử và phê duyệt trên môi trường staging
    - Tất cả test tự động đã pass
    - Kế hoạch deployment đã được review
- **Quy trình triển khai — Rolling Update (thay đổi nhỏ):**
    1. Platform Operator khởi tạo deployment từ CI/CD pipeline
    2. Hệ thống triển khai phiên bản mới theo cơ chế rolling update:
        1. Tạo ECS task mới với phiên bản mới
        2. Đợi task mới pass health check
        3. Chuyển traffic sang task mới
        4. Dừng task cũ
        5. Lặp lại cho đến khi tất cả task được cập nhật
    3. Hệ thống chạy smoke test tự động sau deployment
    4. Nếu smoke test thất bại → Tự động rollback về phiên bản cũ
    5. Ghi Audit Log: “Deployment v{version} completed successfully”
- **Quy trình triển khai — Blue-Green (thay đổi lớn):**
    1. Platform Operator tạo môi trường Green (bản sao của Production)
    2. Triển khai phiên bản mới lên Green
    3. Chạy full test suite trên Green
    4. Chuyển traffic từ Blue (cũ) sang Green (mới) qua CloudFront / ALB
    5. Giám sát Green trong 30 phút
    6. Nếu ổn định: gỡ bỏ Blue
    7. Nếu có sự cố: chuyển traffic về Blue (rollback trong < 5 phút)
- **Quy tắc nghiệp vụ:**
    - BR-14S: Mỗi deployment phải có kế hoạch rollback rõ ràng trước khi bắt đầu
    - BR-14T: Deployment không được thực hiện trong giờ peak (8:00-9:30, cuối tháng)
    - BR-14U: Sau mỗi deployment, hệ thống phải chạy health check toàn diện trong 30 phút
- **Tham chiếu:** Module 13 (NFR-17 — Availability)

### 4.11 Di chuyển schema database cross-tenant (F-1411)

- **Actor:** Platform Operator
- **Màn hình:** Platform Admin Portal > Maintenance > DB Migration
- **Phase:** BF-M09
- **Mô tả:** Khi ứng dụng cần thay đổi cấu trúc database (thêm bảng, thêm cột, thay đổi index), phải áp dụng migration cho tất cả tenant schema đồng thời. Đây là thách thức đặc thù của kiến trúc schema-per-tenant.
- **Điều kiện tiên quyết:**
    - Migration script đã được kiểm thử trên môi trường staging
    - Có kế hoạch rollback cho mỗi migration
- **Luồng xử lý chính:**
    1. Platform Operator upload migration script
    2. Hệ thống phân tích migration:
        - Ước tính thời gian chạy trên mỗi tenant
        - Phát hiện các thay đổi có rủi ro (ALTER TABLE lớn, DROP COLUMN)
        - Kiểm tra tương thích ngược (backward compatibility)
    3. Hệ thống hiển thị kế hoạch migration:
        
        
        | Trường | Mô tả |
        | --- | --- |
        | Số tenant cần áp dụng | Tổng số tenant ACTIVE |
        | Thời gian ước tính / tenant | Thời gian trung bình |
        | Tổng thời gian ước tính | Số tenant x thời gian/tenant |
        | Mức độ rủi ro | Low / Medium / High |
        | Có cần downtime không | Yes / No |
    4. Platform Operator phê duyệt kế hoạch
    5. Hệ thống thực hiện migration theo thứ tự:
        1. Chạy trên 1 tenant “canary” trước
        2. Kiểm tra kết quả trên canary tenant
        3. Nếu thành công: chạy trên các tenant còn lại (có thể song song, giới hạn concurrency)
        4. Nếu thất bại: dừng lại, rollback canary tenant, gửi cảnh báo
    6. Hệ thống ghi kết quả migration cho từng tenant:
        
        
        | Trường | Mô tả |
        | --- | --- |
        | Tenant ID | Mã tenant |
        | Trạng thái | SUCCESS / FAILED / SKIPPED |
        | Thời gian chạy | Giây |
        | Lỗi (nếu có) | Chi tiết lỗi |
    7. Ghi Audit Log: “DB migration {migration_id} applied to {N} tenants. Success: {S}, Failed: {F}”
- **Luồng ngoại lệ:**
    - **E-1411-1:** Migration thất bại trên canary tenant → Dừng toàn bộ, rollback, Platform Operator phải kiểm tra và sửa migration
    - **E-1411-2:** Migration thất bại trên 1 tenant trong batch → Đánh dấu tenant lỗi, tiếp tục các tenant khác, gửi báo cáo
- **Quy tắc nghiệp vụ:**
    - BR-14V: Tất cả migration PHẢI tương thích ngược (backward compatible) để hỗ trợ zero-downtime deployment
    - BR-14W: Migration PHẢI chạy trên canary tenant trước, chỉ tiến hành toàn bộ khi canary thành công
    - BR-14X: Mỗi migration phải có rollback script tương ứng, đã được kiểm thử
- **Tham chiếu:** Module 13 (NFR-12 — Database Isolation)

### 4.12 Quản lý Feature Flag theo tenant (F-1412)

- **Actor:** Platform Admin
- **Màn hình:** Platform Admin Portal > Maintenance > Feature Flags
- **Phase:** BF-M09
- **Mô tả:** Quản lý các cờ điều khiển (feature flag) cho phép bật/tắt chức năng của ứng dụng theo từng tenant hoặc nhóm tenant. Giúp triển khai chức năng mới từng bước, A/B testing, và xử lý sự cố nhanh chóng.

        | Trường | Mô tả |
        | --- | --- |
        | Flag Key | Mã định danh duy nhất (vd: `new_invoice_ui`, `attendance_gps`) |

    - BR-14Y: Feature flag được cache trên Redis, thời gian cập nhật tối đa 30 giây sau khi thay đổi
    - BR-14Z: KILL_SWITCH feature flag phải có hiệu lực ngay lập tức (bypass cache, < 5 giây)
    - BR-150: Mỗi thay đổi feature flag phải được ghi Audit Log đầy đủ (ai, khi nào, thay đổi gì)

### 4.19 Quản lý sự cố và leo thang (F-1419)

- **Actor:** Platform Admin
- **Màn hình:** Platform Admin Portal > Incidents > Incident Mgmt
- **Phase:** BF-M09
- **Mô tả:** Hệ thống quản lý sự cố (incident management) cho phép Platform Admin tạo, theo dõi và xử lý các sự cố hệ thống theo quy trình chuẩn.
- **Luồng xử lý chính:**
    1. Sự cố được tạo từ 2 nguồn:
        1. Tự động: hệ thống phát hiện bất thường và tạo incident tự động
        2. Thủ công: Platform Admin tạo incident khi nhận được báo cáo sự cố
    2. Thông tin incident:
        
        
        | Trường | Kiểu | Bắt buộc | Mô tả |
        | --- | --- | --- | --- |
        | Incident ID | Auto | — | Tự động sinh: INC-{YYYYMMDD}-{seq} |
        | Tiêu đề | String(200) | Có | Mô tả ngắn gọn sự cố |
        | Mức độ | Enum | Có | P1 (Critical) / P2 (High) / P3 (Medium) / P4 (Low) |
        | Trạng thái | Enum | Có | OPEN / INVESTIGATING / IDENTIFIED / MONITORING / RESOLVED / CLOSED |
        | Phạm vi ảnh hưởng | Enum | Có | ALL_TENANTS / SPECIFIC_TENANTS / INTERNAL_ONLY |
        | Tenant bị ảnh hưởng | Multi-select | Không | Danh sách tenant liên quan |
        | Nguyên nhân | Text | Không | Mô tả nguyên nhân (cập nhật khi xác định) |
        | Giải pháp | Text | Không | Mô tả cách xử lý |
        | Người xử lý | Select | Có | Platform Admin / Platform Operator |
        | Thời gian phát hiện | DateTime | Có | Thời điểm phát hiện sự cố |
        | Thời gian xử lý xong | DateTime | Không | Thời điểm xử lý hoàn tất |
    3. Quy trình xử lý theo mức độ:
        
        
        | Mức độ | Định nghĩa | Thời gian phản hồi | Thời gian xử lý mục tiêu |
        | --- | --- | --- | --- |
        | P1 — Critical | Hệ thống ngừng hoạt động hoàn toàn hoặc mất dữ liệu | 15 phút | 1 giờ |
        | P2 — High | Chức năng chính bị ảnh hưởng, nhiều tenant | 30 phút | 4 giờ |
        | P3 — Medium | Chức năng phụ bị ảnh hưởng, ít tenant | 2 giờ | 24 giờ |
        | P4 — Low | Ảnh hưởng nhỏ, có workaround | 24 giờ | 72 giờ |
    4. Leo thang tự động:
        
        
        | Điều kiện | Hành động |
        | --- | --- |
        | P1 chưa phản hồi sau 15 phút | Leo thang lên quản lý cấp cao + gửi SMS |
        | P2 chưa xử lý sau 4 giờ | Nâng cấp lên P1 |
        | Sự cố ảnh hưởng > 50% tenant | Tự động nâng cấp lên P1 |
    5. Khi sự cố được xử lý:
        1. Cập nhật trạng thái → RESOLVED
        2. Nhập nguyên nhân và giải pháp
        3. Hệ thống gửi thông báo cho các tenant bị ảnh hưởng: “Sự cố đã được xử lý”
        4. Tạo post-mortem report (tùy chọn)
- **Quy tắc nghiệp vụ:**
    - BR-167: Mọi sự cố P1 và P2 PHẢI có post-mortem report trong vòng 48 giờ sau khi xử lý
    - BR-168: Toàn bộ lịch sử cập nhật incident được lưu trữ và không thể xóa (immutable timeline)
    - BR-169: Thông tin sự cố gửi cho tenant bị ảnh hưởng chỉ bao gồm mức độ ảnh hưởng và thời gian dự kiến xử lý, KHÔNG bao gồm chi tiết kỹ thuật nội bộ

### 4.21 Health check tự động và tự phục hồi (F-1421)

- **Actor:** Hệ thống (tự động)
- **Màn hình:** Không có màn hình (background process)
- **Phase:** BF-M09
- **Mô tả:** Hệ thống tự động kiểm tra sức khỏe các thành phần và thực hiện các biện pháp tự phục hồi khi phát hiện sự cố.
- **Các loại health check:**
    
    
    | Thành phần | Kiểm tra | Tần suất | Hành động khi lỗi |
    | --- | --- | --- | --- |
    | ECS Task (API) | HTTP GET /health | 30 giây | Restart task, thay thế bằng task mới |
    | RDS Primary | Connection test + simple query | 1 phút | Failover sang standby (Multi-AZ) |
    | RDS Read Replica | Replication lag check | 1 phút | Loại khỏi read pool, cảnh báo |
    | Elasticache | PING + memory check | 1 phút | Failover sang replica node |
    | SQS Queue | Queue depth + oldest message age | 5 phút | Tăng số consumer, cảnh báo |
    | SES | Send quota remaining | 15 phút | Cảnh báo, chuyển sang queue chờ xử lý sau |
    | CloudFront | Error rate check | 5 phút | Cảnh báo, kiểm tra origin health |
- **Cơ chế tự phục hồi (Auto-recovery):**
    
    
    | Sự cố | Hành động tự động | Điều kiện | Giới hạn |
    | --- | --- | --- | --- |
    | ECS task không phản hồi | Restart task | Health check fail 3 lần liên tục | Tối đa 5 lần restart / 10 phút |
    | DB connection exhaustion | Reset idle connections | Connection usage > 90% | — |
    | Redis memory full | Evict expired keys | Memory usage > 90% | — |
    | SQS queue backlog | Scale up consumer tasks | Queue depth > 1000 | Tối đa 10 consumer tasks |
    | High API latency | Scale up ECS tasks | p95 > 5s trong 5 phút | Theo Auto Scaling policy |
- **Quy tắc nghiệp vụ:**
    - BR-173: Health check endpoint (/health) KHÔNG tính vào rate limiting và KHÔNG yêu cầu authentication
    - BR-174: Tự phục hồi chỉ áp dụng cho các sự cố đã biết trước (known failure patterns). Sự cố mới phải leo thang cho Platform Admin
    - BR-175: Mỗi lần tự phục hồi đều phải ghi log chi tiết và thông báo cho Platform Admin (không im lặng)
- **Tham chiếu:** Module 13 (NFR-19 — Graceful Degradation)

### 4.27 Quản lý thử lại job thất bại (F-1427)

- **Actor:** Platform Admin
- **Màn hình:** Platform Admin Portal > Batch > Failed Jobs
- **Phase:** BF-M09
- **Mô tả:** Quản lý các batch job đã thất bại: xem chi tiết lỗi, retry thủ công, hoặc skip tenant lỗi để tiếp tục xử lý các tenant khác.
- **Điều kiện tiên quyết:**
    - Có ít nhất 1 batch job ở trạng thái FAILED
- **Luồng xử lý chính:**
    1. Platform Admin truy cập “Failed Jobs”
    2. Hệ thống hiển thị danh sách job thất bại:
        
        
        | Trường | Mô tả |
        | --- | --- |
        | Job ID | Mã thực hiện |
        | Job name | Tên batch job |
        | Thời điểm thất bại | Timestamp (JST) |
        | Tenant bị lỗi | Tenant cụ thể bị lỗi |
        | Error type | Loại lỗi (DB error, timeout, data error, …) |
        | Error message | Nội dung lỗi chi tiết |
        | Auto-retry count | Số lần đã tự động retry |
        | Trạng thái | FAILED / RETRYING / MANUALLY_RESOLVED / SKIPPED |
    3. Platform Admin chọn job thất bại và thực hiện:
        1. **Retry:** Chạy lại job cho tenant bị lỗi
        2. **Retry All:** Chạy lại tất cả tenant bị lỗi trong job
        3. **Skip:** Đánh dấu tenant lỗi là SKIPPED, bỏ qua
        4. **Investigate:** Xem chi tiết log, stack trace
    4. Khi retry:
        - Hệ thống chạy lại batch cho tenant cụ thể
        - Hệ thống cập nhật trạng thái: RETRYING → SUCCESS hoặc FAILED
        - Gửi thông báo kết quả cho Platform Admin
- **Luồng ngoại lệ:**
    - **E-1427-1:** Retry vẫn thất bại → Ghi log chi tiết, đề xuất Platform Admin kiểm tra dữ liệu của tenant hoặc xử lý thủ công
- **Quy tắc nghiệp vụ:**
    - BR-187: Hệ thống tự động retry tối đa 3 lần với khoảng cách tăng dần (exponential backoff: 1 phút, 5 phút, 15 phút)
    - BR-188: Sau 3 lần auto-retry thất bại, job được đánh dấu FAILED và chờ Platform Admin xử lý thủ công
    - BR-189: Platform Admin có thể retry thủ công không giới hạn số lần, nhưng mỗi lần retry đều được ghi Audit Log
    - BR-190: Skip tenant chỉ được sử dụng khi lỗi là do dữ liệu của tenant, không phải lỗi hệ thống — phải ghi lý do skip

### 4.28 Cấu hình lịch batch job (F-1428)

- **Actor:** Platform Admin
- **Màn hình:** Platform Admin Portal > Batch > Schedule Config
- **Phase:** BF-M09
- **Mô tả:** Cấu hình và quản lý lịch chạy của các batch job, bao gồm thay đổi thời gian chạy, bật/tắt job, và cấu hình tham số chạy.
- **Điều kiện tiên quyết:**
    - Platform Admin đã đăng nhập
- **Luồng xử lý chính:**
    1. Platform Admin truy cập “Cấu hình lịch batch”
    2. Xem danh sách tất cả batch job với lịch hiện tại
    3. Chọn job cần cấu hình:
        
        
        | Trường | Kiểu | Bắt buộc | Mô tả |
        | --- | --- | --- | --- |
        | Trạng thái | Enum | Có | ENABLED / DISABLED |
        | Cron expression | String | Có | Biểu thức cron (vd: `0 2 * * *`) |
        | Timezone | Enum | Có | Mặc định: Asia/Tokyo |
        | Max concurrent tenants | Integer | Có | Số tenant xử lý đồng thời |
        | Timeout (phút) | Integer | Có | Thời gian tối đa cho phép chạy |
        | Retry policy | Enum | Có | NO_RETRY / AUTO_RETRY / MANUAL |
        | Max retries | Integer | Không | Chỉ khi retry policy = AUTO_RETRY |
        | Tenant filter | Multi-select | Không | Chỉ chạy cho các tenant cụ thể (mặc định: tất cả) |
    4. Nhấn “Lưu cấu hình”
    5. Hệ thống cập nhật EventBridge rule tương ứng
    6. Ghi Audit Log: “Batch schedule updated for {job_name}”
- **Tính năng bổ sung:**
    - Chạy thủ công (Manual trigger): Platform Admin có thể kích hoạt bất kỳ batch job nào ngay lập tức
    - Dry run: Chạy thử không áp dụng thay đổi, chỉ báo cáo kết quả dự kiến
    - Lịch sử thay đổi lịch: xem ai đã thay đổi gì, khi nào
- **Quy tắc nghiệp vụ:**
    - BR-191: Batch job KHÔNG được đặt lịch trong giờ peak (8:00-9:30, 17:00-18:00 JST)
    - BR-192: Thay đổi lịch batch job phải có hiệu lực từ lần chạy tiếp theo, không ảnh hưởng job đang chạy
    - BR-193: Tắt (DISABLED) batch job quan trọng (daily_attendance, invoice_generation) phải yêu cầu xác nhận 2 lần

## 7. Tham chiếu kỹ thuật

### 7.1 Hạ tầng AWS liên quan

| Thành phần | Service AWS | Vai trò trong module |
| --- | --- | --- |
| Monitoring | CloudWatch Metrics + Alarms | Thu thập metric, cảnh báo tự động |
| Logging | CloudWatch Logs | Tập trung log từ ECS, RDS, Lambda |
| Scheduling | EventBridge Scheduler | Điều phối batch job theo lịch |
| Queue | SQS + Dead Letter Queue | Xử lý bất đồng bộ, retry logic |
| Storage | S3 + S3 Glacier | Lưu file, archive dữ liệu tenant |
| Cache | ElastiCache (Redis) | Cache feature flag, session, metric tạm thời |
| Email | SES | Gửi thông báo, cảnh báo |
| CDN | CloudFront | Phân phối assets White-label |
| Database | RDS PostgreSQL (Multi-AZ) | Database chính, failover tự động |
| Container | ECS Fargate | Chạy API, hỗ trợ rolling update và auto-scaling |
| Certificate | ACM | SSL cho domain riêng của Provider |

### 7.2 Nguyên tắc thiết kế module

1. **Tenant Isolation:** Mỗi thao tác vận hành phải tôn trọng nguyên tắc tách biệt tenant — dữ liệu, backup, batch job, monitoring đều theo đơn vị tenant
2. **Least Privilege:** Platform Admin chỉ được xem metadata và chỉ số — KHÔNG truy cập dữ liệu nghiệp vụ của tenant
3. **Immutable Audit:** Mỗi thao tác vận hành đều được ghi Audit Log bất biến
4. **Graceful Degradation:** Lỗi của 1 tenant không được ảnh hưởng đến tenant khác và platform tổng thể
5. **Automation First:** Ưu tiên tự động hóa (auto-recovery, auto-retry, auto-alert) trước khi cần sự can thiệp thủ công
